## Open forks in the road

### Fork: the round-2 extension rejects — engine work or accept the drift

Round 2 (`rebuild/POLICY-ROUND-2-REPORT.md`) found that all 29 reviewer rejects are a single phenomenon — ·May's `en-ext-1` baseline-entry extension — and an exhaustive, build-tested lever hunt (25 candidate records across 5 independent lever families, every one scratch-built against a harness proven byte-identical to the oracle; zero cleared the bar) proved that **no closed-vocabulary policy record can honor them**. The wall is a depth-2 left-context predicate the frozen `when:` grammar cannot express (the predecessor's own entry state for the ss03 "pea-may" windows; the predecessor's predecessor for the "it-may" windows), compounded by a token grammar in which `en-ext-1` is un-droppable by any `extend` (always re-emits it), `contract` (coexists without netting), or `refuse` (collapses the whole join).

The two paths:

- **Engine work** — extend the same-seam non-summing rule so it also fires on ·May's _own_ summed exit (the ss03 "pea-may" case), and add a depth-2 left-context predicate (the "it-may" case). Touches `rebuild/pipeline/settle.py`, the rune schema, and tests. The 29 rejects then become honorable. Tracked below as the extension-suppression milestone.
- **Accept the drift** — keep the composed extension everywhere and retire the 29 rejects as taste-accepted divergence.

**Current lean (2026-06-12): accept the drift and postpone the engine work**, on the expectation that a more capable model will make the engine change cheaper and safer to attempt later. The drift is fully documented and recoverable: the 27 + 2 windows are recorded as engine-limited residue in the two reviewed ledger entries (`halves-entry-extension-restored`, `same-seam-extension-non-summing` in `rebuild/m1-divergences.yaml`), the proof and structural analysis are in `rebuild/POLICY-ROUND-2-REPORT.md`, and the verdict pile is `verdicts-08.19.24PM.json`. The lever-hunt evidence is the workflow at `tmp/lever-hunt-wf.js` and its result `tmp/wf2-result.json`; the build harness that produced it is `tmp/scratch_build.py` (parameterized `load_spec(runes_dir, ...)` build into a scratch out-dir).

To revisit: re-read `rebuild/POLICY-ROUND-2-REPORT.md` ("Why it is structurally impossible"), then decide engine-vs-drift. If engine, the work is the extension-suppression milestone below.

**Update (2026-07-03): the ZWNJ-equals-word-boundary rule is ratified and mechanized — no window containing a ZWNJ ever needs a verdict again.** The rule: post-ZWNJ is definitionally word-initial and pre-ZWNJ definitionally word-final, so every segment of a ZWNJ window must render exactly as that segment standing alone, and a ZWNJ unit therefore carries no adjudicable information its boundary-siblings don't (every segment of a length-≤5 window is itself an enumerated window). Motivation: 23% of review units (3,570/15,645) contained a ZWNJ, and the old font's shadow-universe inconsistency made the surface ask the same design question against two different old-font backdrops — producing at least one direct contradiction (u-15220 `·No·It·Day` approved vs u-15230 `·No·It·Day ◊ZWNJ` rejected on identical new ink); an audit of all 44 past ZWNJ rejects confirmed none was about the ZWNJ context itself. The guarantee lives at three levels, two of them permanent vetting code rather than transition ledger: (1) the rune schema no longer admits `is: zwnj` (`boundaryValue` enum, with `test_zwnj_not_addressable_in_when`), so no record can ever distinguish a ZWNJ context from a text edge; (2) `check_split_buffer` + `check_zwnj_structure` in `run_conformance` enforce the rendering half per build over the exhaustive enumeration, now with **zero carve-outs** — the namer dot no longer lowers through a ZWNJ (an `ignore sub periodcentered' uni200C` ahead of the lowering rule in `m1_namer_dot_word_start`; HarfBuzz otherwise skips the default-ignorable ZWNJ in contextual matching), which removed the one codified cross-ZWNJ behavior and let `_slot_signature` drop its dot special-case; (3) the `zwnj-boundary-echo` ledger class absorbs every non-position divergent row containing a ZWNJ at top precedence, subsuming the ZWNJ share of `zwnj-word-initial-unification` / `zwnj-follower-exit-restored` / `zwnj-word-initial-seam-moved` and discharging the seam-moved entry's pending human-verdict pass. Measured on the rebuilt audit (M1.otf recompiled with the dot fix): the gate proves the invariant over 66,045 ZWNJ sequences × 8 configs (528,360 shaping runs, 0 divergences, ~9 s — it now runs in `run_m1.main()` as a hard gate before the oracle, writing `zwnj_equivalence_summary.json`); the blanket took 17,995 rows / 3,415 units, the three subsumed `zwnj-*` classes came back **empty** (no namer-dot shadow remainder exists in the current alphabet; entries kept at count 0 for the documentation trail), the 188 position-only ZWNJ rows stay on `kern-channel-out-of-scope`, `multi_matched` stayed 0 (the ss10 predicate now excludes ZWNJ rows so the partition stays exact), and **zero UNMATCHED rows contain a ZWNJ** — a preview surface rebuild (`tmp/review-preview`, 15,808 units) confirms every verdict family is ZWNJ-free (the 41 unverdicted regrouping-floor-drift stragglers included). What this forecloses, by design: the old font's post-ZWNJ renderings are unrecoverable — preferring an old ZWNJ-only behavior means changing the word-initial form, which changes both. The dot-through-ZWNJ divergence from the old font is intended and rides the blanket class; a `00B7 200C E670` row joined the CoreText smoke set. Existing ZWNJ verdicts in the masters become inert history. One wart for the phase-end rebuild: `rebuild.review.build` does not delete orphaned class shards, so clear the target `units/` dir (or prune shards for classes absent from the manifest) when rebuilding in place. **Same-day follow-up: the rule now covers space too — every run-splitting boundary equals a text edge.** The gate was generalized to split on every `splits_runs` boundary token (renamed `run_boundary_equivalence`, writing `boundary_equivalence_summary.json`; 110,726 boundary-containing sequences × 8 configs = 885,808 shaping runs, 0 divergences, ~15 s per build), and `is: space` was removed from the schema's `boundaryValue` enum alongside `is: zwnj` (`test_run_splitting_boundaries_not_addressable_in_when`), so no rune record can render a pre- or post-space context differently from a text edge either. `is: namer-dot` deliberately survives — the dot does not split runs (position stays medial per BETTERRUNESCHEMA R32) and keeps its own semantics. Note the asymmetry left standing: space windows still go through the ordinary ledger/review machinery (no space analogue of `zwnj-boundary-echo` was authored), because the old font kerns across spaces and the review dedup for space windows was not part of this ratification — extend the blanket to space only with a fresh decision.

- **ss03 full-size ·Tea when a baseline-joiner follows** — the clarified meaning of ss03: under ss03 an x-height exiter joins ·Tea at the x-height, and ·Tea should take its **full-size** form (entered at the x-height, exiting at the baseline) **when the letter after ·Tea joins ·Tea at the baseline** — only the full bar descends far enough to offer that exit. In isolation the **half** stub is correctly chosen, so the distinction is invisible without a baseline-joining follower. This is a **gap**: today `qsTea.yaml`'s `full` stance unlocks an x-height entry only for `left: qsFee` (ss03) / `left: qsI` (ss02); the broad exiter set unlocks only the `half` stance (no baseline exit), so e.g. `·May·Tea·Utter` renders the half stub and ·Utter fails to baseline-join. Honoring it is a **deliberate divergence from the legacy oracle** (legacy has only `half_entry_xheight_ss03` for the broad set plus `entry_xheight_after_fee` for ·Fee), so it needs the oracle/review-surface re-blessing path, not a parity fix. The docs (README ss03 blurb, `rebuild/script.yaml` description, `FEATURE_DESCRIPTIONS`) were corrected on 2026-06-27 to describe this intent; the engine change is deferred. **Mechanism when built:** add one right-gated unlock to `qsTea.yaml`'s `full` stance mirroring the half stance's broad left-set, gated `right: {class: can-enter-at-baseline}` (the predicate class already exists in `script.yaml`); `order: [full, half]` then yields "full when a baseline-joiner follows, half otherwise" (in true isolation the right-class predicate fails against the EDGE token, so half is still chosen), and the full stance already permits the `{entry: x-height, exit: baseline}` pairing. **Caveat the frozen `when:` grammar cannot fence:** `can-enter-at-baseline` is a _static_ capability predicate, so it over-triggers on followers that have a selectable baseline entry but do not actually accept ·Tea's baseline join (qsTea, qsIt, qsPea, qsNo) — for those, full's _non-joining_ cell would win the order tiebreak and render a full bar where the half stub is wanted, so the build also needs a suppressing `refuse` on full's non-joining cell for that follower set, or an explicit decision to accept the full bar there.

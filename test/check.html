Check — before/after

Side-by-side rendering harness. Two auto-generated sections cover the whole page: one diffs every multi-letter Quikscript run in the test corpus between the snapshot under test/before/ and the live build under test/; the other lists every short sequence whose adjacent non-joining pair changes shape between single-buffer and split shaping.

Workflow:

  1. On the baseline you want to compare against (typically master), run make snapshot-before. It builds the fonts and copies all six OTFs into test/before/ (gitignored).
  2. Make your code or YAML changes on a branch.
  3. make check-html rebuilds the live OTFs and regenerates this file end-to-end via tools/build_check_html.py.
  4. Reload this page. Codepoint references live in reference/csur/index.html; the family-to-codepoint map is in postscript_glyph_names.yaml.

Auto-generated: corpus render diffs

Every multi-letter Quikscript run harvested from test/the-manual.html, test/index.html, and test/extra-senior-words.html whose Senior-Regular render differs between test/before/ and the live build. A render is the per-glyph tuple (glyph name, outline hash, x_advance, x_offset, y_offset), so this surfaces GSUB changes (different variant chosen), GPOS changes (cursive/kerning shifts), and outline edits to same-named glyphs.

Each row's label names the change in glyph terms (e.g. qsTea → qsTea.exit-xheight). Scan for sequences you weren't intending to touch — those are the regressions.

Auto-generated: isolation leaks

Sequences whose adjacent pair does not cursive-attach but whose chosen glyphs differ when the pair is shaped together versus split into independent buffers. These are the cases that currently require |?| in data-expect; visually inspect each row to decide whether the cross-break shape change is cosmetic or a real bug. Generated with --max-len 3.

Columns: the middle column shapes the whole sequence as a single buffer (what you get in real text); the right column splits the sequence at the leaky break into two display: inline-block halves so HarfBuzz shapes each side independently. If the two columns look identical, the leak is purely a glyph-name signature change with no visible effect; if they differ, decide whether the in-context shape is intended.

Each row is tagged same or diff based on whether the in-context buffer and the concatenation of the two independently-shaped halves produce the same per-glyph (pixels, absolute origin) sequence. Comparing absolute origins (not just pixels) catches cursive-positioning leaks: e.g. qsIt vs qsIt.exit-xheight have identical bitmaps but the latter's exit anchor pulls the next glyph leftward via GPOS curs. Reach for the diff rows first when hunting real bugs; same rows are typically after-tall-style trims whose only effect is the glyph-name signature.

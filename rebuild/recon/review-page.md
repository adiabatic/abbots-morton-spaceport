# Recon: page patterns and frontend rules for the §11 review surface

Source survey for the `rebuild/out/review/` migration-baseline-diff app (design §11). Everything here was read directly from the repo on 2026-06-10; line references are to current `master` files. The new generator is `rebuild/` code — imitate the patterns below, never import `tools/build_check_html.py`.

## 1. Global frontend rules (`~/.agent-config/frontend-design.md`)

The user's global frontend workflow doc is short (61 lines). The load-bearing rules, verbatim:

> Build distinctive frontends with intentional aesthetic choices, not generic "AI slop" patterns. Pick a clear conceptual direction and execute it with precision — the key is intentionality, not intensity. Match implementation complexity to the vision: maximalist designs need elaborate animations and effects; minimalist designs need restraint and precision.

Design thinking, before coding:

> - **Purpose**: What problem does this interface solve? Who uses it?
> - **Tone**: Pick a specific direction (brutally minimal, maximalist chaos, retro-futuristic, organic/natural, luxury/refined, playful/toy-like, editorial/magazine, brutalist/raw, art deco/geometric, soft/pastel, industrial/utilitarian, etc.). Use the list for inspiration, but design something true to the chosen direction rather than imitating a reference.
> - **Differentiation**: What's the one detail someone will remember after closing the tab?

The avoid list (verbatim, these are the named failure modes):

> - Hover effects on non-clickable elements
> - Overused font families (Lato, Roboto, Arial, system fonts)
> - Fonts that lack multiple weights or a built-in italic/oblique style
> - Cliched color schemes (particularly purple gradients on white backgrounds)
> - Predictable layouts and component patterns
> - Cookie-cutter design that lacks context-specific character

Accessibility (best-effort WCAG 2.1 AA): semantic HTML elements (`nav`, `main`, `button`, `dialog`); 4.5:1 / 3:1 contrast; "All interactive elements keyboard-accessible with visible focus indicators"; ARIA only when semantic HTML isn't sufficient; "Support `prefers-reduced-motion` with animation fallbacks"; logical heading hierarchy.

Practical reading for this tool: it is an internal triage instrument used by one person for hours at a stretch — the right tone is utilitarian-precise, the differentiator is the one-keystroke verdict flow, and the existing `site/` pages (Seravek/Corbel chrome, `light-dark()` everywhere, Menlo for code) already constitute a coherent house style worth staying consistent with rather than reinventing. Note the existing site pages deliberately use system fonts for chrome; consistency with the established house style trumps the "overused font families" bullet here, since the content typography IS the font under test.

Repo-level HTML/CSS/JS rules (from `CLAUDE.md`, binding): nested CSS over flat; `for-of` over `.forEach()`; modern range syntax `(width > 40em)`; view state in the URL, not localStorage.

## 2. The proven tester-page chrome: `site/check.html` + `tools/build_check_html.py`

`site/check.html` is fully generated by `tools/build_check_html.py` (1,971 lines) — Python f-string assembly of one self-contained HTML file: a `_render_page()` template taking pre-rendered section strings, a `_PAGE_CSS` constant, and `_COPY_CODEPOINTS_SCRIPT` / `_VERDICT_SCRIPT` constants holding the inline `<script>` blocks. No templating library, no client-side data fetch — all rows are baked into the HTML at generation time. This is the approach to imitate.

### Dual @font-face before/after wiring (check.html:15-61)

```css
@font-face {
  font-family: "AMS Sans Senior — Before";
  src: url("before/AbbotsMortonSpaceportSansSenior-Regular.otf") format("opentype");
  font-weight: 400;
  font-style: normal;
}
@font-face {
  font-family: "AMS Sans Senior — After";
  src: url("AbbotsMortonSpaceportSansSenior-Regular.otf") format("opentype");
  ...
}
:root {
  --font-size: 88px;
  --before-font: "AMS Sans Senior — Before", "Departure Mono", monospace;
  --after-font: "AMS Sans Senior — After", "Departure Mono", monospace;
}
```

Rows then carry `.qs.before { font-family: var(--before-font) }` / `.qs.after { font-family: var(--after-font) }`. The "before" OTFs live in a gitignored snapshot directory (`site/before/`, populated by `make check-html-before`); the "after" OTFs are the live build. For the review app the analogous pair is baseline-branch OTFs vs. rebuild-pipeline OTFs, both copied under `rebuild/out/review/` (gitignored) so the page is self-contained. Crucial rendering detail on every Quikscript sample: `-webkit-font-smoothing: none; font-smooth: never;` — the font is a pixel font and must not be antialiased — while prose chrome re-asserts `subpixel-antialiased`.

### Checkered background (check.html:200-211)

The proven CSS, applied to each rendered sample cell so whitespace/advance changes are visible against a reference grid:

```css
.render-diffs .qs.before,
.render-diffs .qs.after {
  --grid-color: light-dark(rgba(0, 0, 0, 0.05), rgba(255, 255, 255, 0.06));
  background-image:
    linear-gradient(45deg, var(--grid-color) 25%, transparent 25%, transparent 75%, var(--grid-color) 75%),
    linear-gradient(45deg, var(--grid-color) 25%, transparent 25%, transparent 75%, var(--grid-color) 75%);
  background-size: 16px 16px;
  background-position: 0 5.6px, 8px 13.6px;
}
```

The odd `background-position` y-values phase-align the 16px checker to the font's pixel grid at `--font-size: 88px` (88px / 11 rows = 8px per font pixel; 5.6px offsets the checker to the baseline). If the review page uses a different sample size, recompute the phase rather than copying the magic numbers.

### Row structure (corpus render diffs / leak rows)

Every row is a CSS grid: `grid-template-columns: 32ch 1fr 1fr` — label column, then the two rendered columns — with `align-items: baseline`, a thin bottom border, and sticky `.col-headers` sharing the same grid template. The label cell stacks: optional open-in-tables icon links, a `.visual-tag` classification chip (`diff`/`benign`/`bad`, colored by `data-visual` attribute on the row), the human-readable sequence (`·Pea ·Zoo | ·Tea ·Oy (qsZoo.en-con-2 → qsZoo)`), and a `.codepoints` strip with a copy button plus `<code data-letters="·Pea·Zoo·Tea·Oy">U+E650 U+E65B …</code>`.

The copy button copies a **prompt preamble** ("I'm looking at site/check.html — specifically, U+E650 … (·Pea·Zoo·Tea·Oy). ") to the clipboard — explicitly designed for pasting into an agent conversation. Worth carrying over with the path updated.

### How pairs are highlighted / verdicts marked

check.html's depth-4 triage section (the closest existing ancestor of the §11 verdict UI) marks verdicts visually rather than highlighting the pair per se: a "broken" verdict overlays a red X over the in-context preview via a `::after` with two `linear-gradient` diagonals (check.html:517-526); "in-context better"/"halves better" ring the preferred preview with `outline: 2px solid` green (`mark-good`). Row-level state: `.row.has-verdict { background: light-dark(#eef4ff, #1b2740) }`. The §11 requirement "the pair under review unmistakably highlighted" goes further than anything existing — the closest pattern for highlighting a sub-span inside a shaped run is the leak rows' split rendering (below) where each half is its own `<span class="half">`; for highlighting the changed pair inside a continuous run, the workable approach is wrapping the pair's codepoints in a styled span **only when the pair joins to nothing across the span boundary**, otherwise the wrap breaks shaping — so prefer a marker rendered outside the text (underline bar positioned under the pair via a measured offset, or a second dimmed/highlighted copy of the run) over inline span-wrapping.

### Split shaping (the inline-block trick)

To render "halves shaped separately," each half goes in `display: inline-block` spans inside one container; HarfBuzz/the browser shapes each inline-block independently:

```html
<div class="qs in-context">&#xE650;&#xE65B;&#xE652;&#xE679;</div>
<div class="qs isolated"><span class="half">&#xE650;&#xE65B;</span><span class="half">&#xE652;&#xE679;</span></div>
```

### ZWNJ-containing strings

ZWNJ is emitted as a literal `&#x200C;` entity inside the rendered run (so the font's real `uni200C`-keyed rules fire — e.g. check.html:1021 `&#xE676;&#x200C;&#xE666;&#xE662;`), and made _visible to the human_ only in the label, as the token `◊ZWNJ` (and space as `␣`): `_BOUNDARY_TOKEN_LABEL = {"space": "␣", "ZWNJ": "◊ZWNJ"}` (build_check_html.py:568). All Quikscript text is emitted as numeric character references via `_entity_for(cp)` / `_entities(text)` — never raw PUA characters in the HTML source — which keeps the generated file greppable and diff-readable.

### Per-ss font-feature-settings

Two proven patterns in the existing site:

- Static, attribute-keyed (the-manual.html:96-118), matching the corpus's `data-stylistic-set` attribute directly — best fit for the review page since each diff row carries a fixed configuration:

  ```css
  [data-stylistic-set~="02"] { font-feature-settings: "ss02" 1; }
  ```

  Multi-set rows need combined rules (`font-feature-settings` doesn't compose across selectors — later declarations replace, not merge), so generate one rule per configuration token actually present in the data (e.g. `[data-config="ss02+ss05"] { font-feature-settings: "ss02" 1, "ss05" 1; }`).
- Dynamic, variable-keyed (kerning.html:13,137,1044): `font-feature-settings: var(--font-features)` plus JS `document.documentElement.style.setProperty("--font-features", caltOn ? "normal" : '"calt" 0')` — the pattern for a page-wide calt/feature toggle button.

### The verdict-collector JS (check.html:1118-1200)

The existing triage loop, worth extending rather than redesigning: a `Map` from row group → verdict string; preset verdict buttons (`data-verdict` holds the exported statement, `data-mark` the visual treatment) plus a free-text `.verdict-custom` input per row; a sticky panel with a readonly `<textarea class="verdict-output">` that rebuilds on every change (one line per verdict: `` `${group.dataset.seq} => ${verdict}` ``), a live count, and a "Copy all verdicts" button. Clicking an active button retracts. Event handling is fully delegated (two listeners on the section, `event.target.closest(...)`), `for-of` throughout. What it lacks vs. §11: keyboard verdicts, auto-advance, persistence (verdicts vanish on reload — the URL-state rule plus the export files should fix that), and structured export (it exports prose lines, not draft pins).

### What in build_check_html.py is worth imitating (not importing)

- One program, one file out; `--out` argument; `main()` prints `Wrote <relpath>` to stderr.
- Sections are independent `_render_X_section()` functions returning HTML strings; the page template takes them as parameters; missing inputs render an explanatory `.snapshot-missing` paragraph instead of failing.
- All shaping/classification happens in Python at generation time; the browser only renders and collects verdicts.
- CSS/JS as module-level Python string constants, kept together with the row-emitting functions that produce the matching markup.
- Deterministic sort keys for rows (`_leak_sort_key`, `_diff_sort_key` — code-point order) so regenerated pages diff cleanly.
- Self-documenting page: the workflow `<ol>` is part of the template, so the page explains its own regeneration commands.
- `html.escape(..., quote=True)` on every attribute payload; data for JS rides in `data-*` attributes, not inline script literals.

What NOT to imitate: its sys.path hacking into `test/` and `tools/` is tolerated legacy; rebuild code already has its own import discipline (`rebuild/validation/pins.py` wraps the one cross-tree import in `_import_test_shaping()`).

## 3. Interaction patterns: `site/tables.html` and `site/kerning.html`

### tables.html — state in the URL (the pattern the user explicitly likes)

View state lives in `location.hash` as `URLSearchParams` (`tables.html#letter=See&col=Ye&row=Fee`):

- `parseHash()` (tables.html:291-321) reads the hash into a state object with defaults for anything absent.
- `writeHash()` (tables.html:323-329) serializes state back: `location.hash = params.toString()`.
- A single `applyHashState()` renderer (tables.html:501-530) is the only thing that mutates the DOM from state; it diffs against `renderedLetterIndex` / `renderedStripKey` / `renderedCellKey` memos so re-entrant hash changes don't rebuild what's already rendered. `window.addEventListener('hashchange', applyHashState); applyHashState();` boots it.
- All user actions write the hash, never the DOM directly (e.g. the letter `<select>`'s change handler just calls `writeHash`), making every view bookmarkable/linkable — check.html's open-in-tables links deep-link straight into a cell, which flashes (`flash-cell` keyframes) and scrolls into view.
- `updateTitle()` mirrors state into `document.title`.

One caveat: tables.html keeps its _highlights_ in localStorage (`ams-joins-highlights`) — that part predates the user's "state in the URL, not localStorage" preference and is the part NOT to copy. For the review app: filter/sort/cursor position belong in the URL; verdicts-in-progress are real work product and belong in the export channel (and plausibly a server-side save, since the new server is ours to write), not localStorage.

### Keyboard handling

The existing site is thin here: `shared.js` exports `initEscapeToClear(clearFn)` — one document-level `keydown` listener checking `e.key === 'Escape'` (shared.js:309-312). That's the entire precedent, so the §11 one-key home-row verdict flow (with auto-advance) is new construction. Pattern to follow: one delegated document-level `keydown` handler, guarded by `if (event.target.closest('input, textarea')) return;` so typing a per-row note never triggers a verdict, mapping home-row keys to the same code path the clickable verdict buttons use. Keep buttons as real `<button>` elements for the accessibility rules (focus indicators, keyboard accessible) and treat the home-row keys as accelerators over them.

### Mouse conventions worth keeping

kerning.html establishes click = decrement / right-click (`contextmenu`, with `preventDefault`) = increment for value cells, and shift-click or badge-click to open a drill-down panel (kerning.html:993-1010); tables.html uses right-click on a cell to copy its dotted name with a positioned popover toast (`popover="manual"`, `showPopover()`, with a `color-scheme: light dark` re-assert because the `[popover]` UA rule resets it — tables.html:199-209). Both pages debounce expensive recomputation (kerning.html re-parses its textarea on a 150 ms `setTimeout` after `input`).

### Batching/pagination of many rows

There is no existing pagination precedent: tables.html builds a full 45×45 `<table>` (~2,000 shaped cells) synchronously in `buildTable()` and relies on sticky headers plus browser scrolling; check.html bakes every row into static HTML; kerning.html likewise. The pages stay responsive because rendering shaped text is cheap once `@font-face` is loaded. §11 says "in batches of hundreds" against 15,528 divergence rows — so the new thing to build is batching at the _generation_ layer: emit one static page per batch (or per divergence class) with prev/next links and the batch identity in the URL, rather than a single 15k-row DOM plus client-side virtualization. Static per-batch pages keep the proven everything-baked-in model, keep each page diffable, and let verdict exports be per-batch files. The m1 ledger's 15 divergence classes are the natural top-level grouping; check.html's `details.collapsible` fold chrome (with counts in the `<summary>`) is the proven way to subdivide within a page.

## 4. Serving: `tools/serve.py` and the port-7294 sibling

`tools/serve.py` is 33 lines: `livereload.Server` (already a dependency, `pyproject.toml:16` `livereload>=2.7.1`) over `site/` on port 7293, with a `NoCacheStaticHandler` subclassing tornado's `StaticFileHandler` to set `Cache-Control: no-store` (essential — browsers aggressively cache OTFs, and a stale cached font silently invalidates every visual judgment), and `server.watch()` globs for `**/*.html`, `**/*.css`, `**/*.otf`, `**/*.js`. `make serve` runs it.

Decision: yes, a small sibling script — `rebuild/serve_review.py` mirroring `tools/serve.py` line-for-line with `ROOT / "rebuild" / "out" / "review"` as the served root and `PORT = 7294`, run as `uv run python rebuild/serve_review.py`. Do not touch the Makefile (hard rule); document the command on the generated page itself, check.html-style. Keep the same four watch globs so editing the generator and regenerating triggers a live reload while 7293 serves `site/` unchanged. If verdict persistence ever needs a POST endpoint, livereload's tornado app can mount handlers, but the v1 contract (copy-out / file export) doesn't require it — start static.

## 5. The data-expect minilanguage: what a drafted pin must look like, and how to validate it

References: `doc/data-expect.md` (the spec), `test/test_shaping.py` (the parser + runner), `test/quikscript_shaping_helpers.py` (`_assert_expect_any`), `rebuild/validation/pins.py` (precedent for importing the parser read-only from rebuild code).

### Pin grammar (what the thumbs-up draft must emit)

A pin is a `data-expect` string over glyph tokens joined by connection operators:

- Glyph tokens: `·LetterName` (maps to `qsLetterName` via `_letter_to_qs`; specials: `·-ing` → `qsIng`, `·J'ai`/`·J’ai` → `qsJai`), `\X` literal character (resolved through `postscript_glyph_names.yaml` or `uniXXXX`), `◊space` / `◊ZWNJ` boundary tokens. The parser's token regex (test_shaping.py:86-93): letter name, optional `+`/`+?`/`+|` ligature partner, optional dot-suffixed variant assertions.
- Variant assertions: `.alt` / `.half` (stable traits), `.!alt` / `.!half` (negated), `.∅` (exact bare glyph, combinable with nothing else), plus compat-only modifiers (`en-y0`, `ex-y5`, `noentry`, `extended`, …) matched against compiler-provided `compat_assertions`, not name substrings. **Drafted pins should prefer no variant assertions at all** — "Whole-word assertions remain the preferred cheap lock" (design §10.5), and bare tokens accept any variant.
- Connections between tokens: bare whitespace = join at unasserted height; `~x~` y=5, `~b~` y=0, `~t~` y=8, `~6~` y=6; `|` break (which **also** triggers the break-isolation invariant — split-buffer re-shape must match); `|?|` break without the isolation invariant (for sanctioned cosmetic cross-break influence); `?` connection unasserted. Ligatures: `·Day+Utter` must-ligate, `·Day+?Utter` may-ligate (connection unasserted if separate), `·Day+|Utter` may-ligate (break asserted if separate).
- Example of a full drafted whole-word pin: `·Bay ~b~ ·Roe | ·Day+?Utter ~x~ ·Low`.

A pin row in the corpus is an element (`<td>`, `<span>`, or `<dd>` — `_DataExpectCollector._TAGS`) whose text content is the actual codepoint string and whose `data-expect` attribute is the assertion (`data-expect-noncanonically` for joins not found in Read's manual — the right attribute for review-tool drafts covering sequences outside the Manual, same parser and semantics). **Duplicate discipline** (doc/data-expect.md §Duplicates): if the same text content already carries a `data-expect` anywhere in the corpus under the same feature context, the draft is redundant — the export should flag rather than emit it.

### ss scoping

Feature context is part of the test case and rides on HTML structure, not in the expect string: a `data-stylistic-set="02"` attribute on the pin element itself (collected as the cell's `stylistic_set`), or an inner `<span data-stylistic-set="02 05">…</span>` that starts a new run with `features = {"ss02": True, "ss05": True}` (test_shaping.py:303-309 — note `ss.zfill(2)` and space-separated multi-set values). A `<span class="force-junior">` run switches to the Junior font (connections suppressed there). So a drafted pin for an ss-scoped divergence row must carry the configuration as an attribute alongside the expect string — the review export needs (text, expect, stylistic_set, target file) tuples, not bare strings. Design §10.5 lists "in-string stylistic-set scoping for single connections" as a planned backward-compatible _addition_, not something the parser supports today — do not draft it yet.

### Validating a drafted pin programmatically

Two stages, both invokable from rebuild code (requires the built OTFs under `site/`; follow `rebuild/validation/pins.py:30-38`'s `_import_test_shaping()` pattern — put `ROOT / "test"` on `sys.path` lazily, import read-only):

1. **Syntax**: `test_shaping.parse_expect(raw)` returns `(tokens, connections)` or raises `ValueError` with position info. Cheap, no font needed — run on every keystroke-drafted pin before offering it.
2. **Semantics against the live font**: the real runner is `test_shaping.run_shaping_test_runs(fonts, anchor_maps, runs, expect_str, base_potential_entries=…)`, which shapes each run, tries every maybe-ligature interpretation, checks glyph identity/variants/connections, and (senior only) enforces the break-isolation invariant for every `|`. The ready-made wrapper is `quikscript_shaping_helpers._assert_expect_any(text, [expect])` (quikscript_shaping_helpers.py:234-257): it builds the senior shaping environment once (`@cache`d `_senior_shaping_env()`), wraps the text as a single senior run, and raises `AssertionError` with the full per-candidate diagnostics on failure. Validating a single drafted pin is exactly `_assert_expect_any(text, [drafted_expect])` in a try/except. Build `text` from families with `quikscript_shaping_helpers._qs_text(*families)` (accepts `qsX` names plus `space`/`ZWNJ` tokens). For ss-scoped pins, call `run_shaping_test_runs` directly with `features={"ss02": True}` since `_assert_expect_any` has no features parameter.

Caveat for the M1 use case: validation runs against whichever font is at `site/AbbotsMortonSpaceportSansSenior-Regular.otf` (module constant). A thumbs-up on a divergence row approves the _rebuild_ behavior, so the drafted pin will FAIL against the old-pipeline font whenever the divergence is a genuine behavior change — that's expected and is the pin doing its job at cutover. The export should validate drafts against the rebuild-built font and record validation status per draft; swapping the target font means either monkeypatching `FONT_PATHS`/`FONT_PATH` before import-time caching bites, or (cleaner) reusing `rebuild/validation/shaping.Shaper(font_path)` for shaping and the parser only for syntax. Decide in design, not ad hoc.

### The any-of channel

`_assert_expect_any(text, expects)` is today a Python-test-only construct (`test/test_calt_regressions.py` uses it ~40×): try each candidate `data-expect` string in order, pass on the first that matches, else raise listing every candidate's failure. Design §10.5 plans "an any-of connective at the corpus layer (promoting the Python-only `_assert_expect_any`)" — not yet implemented in the corpus collector.

So a review-tool "fine either way" verdict should export a structured any-of record that today lands as a generated Python test (the only executable channel that exists), and is shaped so it can later migrate to the corpus syntax mechanically:

```yaml
text: "qsPea qsOwe qsNo"          # families, _qs_text-ready (or the literal codepoint string)
features: {}                       # or {ss02: true}
any_of:
  - "·Pea | ·Owe.ex-y5 ~x~ ·No"   # the rebuild behavior
  - "·Pea ~x~ ·Owe ~x~ ·No"       # the baseline behavior, also acceptable
provenance: "m1 divergence-audit row <id>, verdict: fine-either-way"
```

Each `any_of` entry must individually satisfy `parse_expect`; the pair (typically the row's before-behavior and after-behavior, both rendered as full expect strings by the generator so the human never writes syntax) is what "fine either way" means. The matching executable form is one `_assert_expect_any(_qs_text(...), [...])` call.

## Decisions embedded above (surface for confirmation if contested)

- Static per-batch generated pages over client-side virtualization for the 15,528 rows, grouped by divergence class.
- `rebuild/serve_review.py` as a livereload mirror of `tools/serve.py` on 7294; no Makefile change; start static (no POST persistence).
- Verdict-in-progress state: URL for view state, exported files for work product; no localStorage.
- Drafted pins: bare whole-word tokens (no variant assertions) by default; `data-expect-noncanonically` for non-Manual sequences; ss scope as attribute data alongside the string; validate syntax with `parse_expect` always, semantics against the rebuild font.
- Any-of verdicts export as structured records realized as `_assert_expect_any` tests until the corpus-layer connective exists.

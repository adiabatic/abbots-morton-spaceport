# Treaty-diff review surface

## TL;DR

The review surface is built, tested, and ready for human triage. `rebuild/review/` (committed-shape source) generates a self-contained static app under the gitignored `rebuild/out/review/`: review units covering the M1 divergence rows from `rebuild/out/m1/divergence-audit.tsv`, sharded by the ledger classes in `rebuild/m1-divergences.yaml`. Units are machine-approved at build time when both fonts render them pixel-identically once kerning is neutralized — `rebuild/review/ink.py` shapes every unit in both shipped fonts under every config in its set (uharfbuzz, with `kern` disabled on both sides), records each glyph’s outline with fontTools’ `DecomposingRecordingPen`, translates it by the cumulative advance plus per-glyph offsets, and compares the sorted placed pieces; only glyph names differ on those, so no human verdict is meaningful. Every unit renders side-by-side in the old Senior Sans and the M1 mini-font via dual `@font-face`, with the divergent pair highlighted, correct per-row `font-feature-settings`, visible ZWNJ boundary marks, one-key home-row verdicts (`a`/`s`/`d`/`f`), and per-unit notes. Config chips are inert labels backed by build-time `render_groups` (whole-unit verdicts only); each unit shows a single inert `config_note` badge only when its config set is non-trivial (ss03-gated, ss03-excluded, or ss10-only). Every row carries an always-visible one-line explain summary plus a Why? button. Rejecting is a two-step verdict — `s` opens a popup offering `s` no comment plus canned-note keys, with `Escape` canceling. The export loop closes: a downloaded `verdicts.json` converts to one triage YAML whose pins parse with the repo’s real `parse_expect`, whose policy edits name resolvable `glyph_data/runes/*.yaml` keypaths and provenance, and whose any-of drafts feed the any-of channel.

### Commands

```sh
make review-build                                     # regenerate rebuild/out/review/
make review-serve                                     # serve it on http://localhost:7294/
uv run python -m rebuild.review.build                 # the same build, spelled out
uv run python -m rebuild.review.serve                 # the same server, spelled out
uv run python -m rebuild.review.export tmp/verdicts.json --out tmp/review-triage.yaml
```

The Makefile targets are `review-build` / `review-serve` because plain `review` was already taken by the scoped-anchor-selector review page.

Port 7294 deliberately coexists with `make serve` (`tools/serve.py`, port 7293), so both can run simultaneously. Start the server with the serve command above.

## Quick start for triage

### Kerning is disabled

Both columns render with kerning off (`font-kerning: none` on the sample cells, composing with each row’s `font-feature-settings`), and every build-side comparison — the ink-identity census, highlight x-ranges, boundary-mark positions, pin semantics — shapes with `kern: False` in both fonts. The rebuild pipeline deliberately has no kerning until the §12 milestone, so the old font’s kern feature is pure noise in before/after comparisons; kern differences get their own review when that milestone lands. This is a no-op on the after font today, but the rule is explicit so it survives §12.

### Keyboard map

| Key      | Action                                                                                                                                       |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `a`      | Skip + auto-advance                                                                                                                          |
| `s`      | Reject (want the old behavior back) — opens the follow-up popup: `s` no comment, `a` / `f` canned notes; `Escape` cancels. Then auto-advance |
| `e`      | Identical — the highlighted portion looks identical (the flagged difference is invisible) + auto-advance                                     |
| `d`      | Fine either way (any-of channel) + auto-advance                                                                                              |
| `c`      | Neither — both behaviors look wrong (follow-up) + auto-advance                                                                               |
| `f`      | Approve (the new behavior is right) + auto-advance                                                                                           |
| `u`      | Undo last verdict                                                                                                                            |
| `n`      | Focus the note input for the current unit (`Enter` saves and advances)                                                                       |
| `g`      | Approve the whole group under the cursor                                                                                                     |
| `x`      | Toggle the explain panel (same as the row’s Why? button)                                                                                     |
| `↓` `↑`  | Move cursor without verdicting                                                                                                               |
| `k` `i`  | Same as `↓` `↑` — next / previous unit                                                                                                       |
| `[` `]`  | Previous / next class (toolbar arrows page batches)                                                                                          |
| `?`      | Help overlay                                                                                                                                 |
| `Escape` | Blur input / close overlay (or cancel the reject popup)                                                                                      |

The four main verdict keys sit on the left home row: `a` skip, `s` reject, `d` fine-either-way, `f` approve. Rejecting is a two-step: `s` (or the Reject button) opens a small popup anchored to the row’s verdict buttons, and a second key records the reject — `s` with no comment, `a` with the canned note “the old way seems nicer to write out by hand”, `f` with “the new way is broken” (a canned note overwrites whatever was in the note field), `z` with “new way has a worse-looking extension/contraction”, `x` records the reject and drops you straight into the note field for a custom comment; `Escape` or a click anywhere outside cancels with no verdict, and every other shortcut is suppressed while the popup is open. To clear an erroneous verdict entirely, press `Backspace` (or `Delete`) on the unit — pressing its active verdict key again also toggles it off; `u` undoes the most recent verdict from anywhere. The fifth verdict, `c` neither, means both behaviors look wrong — the unit needs follow-up authoring work rather than a pick. The sixth, `e` identical, means the highlighted portion looks identical — the reviewer cannot see the flagged difference, which is signal for the ink-comparator and highlight tooling rather than for either font. In the verdict grid the Identical button sits in row 1’s third column with Either and Neither stacked directly below it. `k` and `i` are home-row aliases for `↓` and `↑`. Shortcuts are suppressed while a note input is focused; `Escape` gets you back out. A verdict always covers the whole unit — every config in its config set (most units show no config marker because they diverge under every non-ss10 config; when a small badge like “only when ss03 is on” is present, hover it for the full list). Approving a unit draws a green outline around its after sample; rejecting overlays a red X on it; either outlines both samples green (both are fine); neither overlays the red X on both (both are wrong) — all four visuals follow the row’s recorded verdict, so they survive re-render and import, and clear on undo.

### Reading the explain panel

Every row carries an always-visible one-line summary under the renderings — what the new pipeline chose at the primary divergence and the single deciding record, in rune-name notation, e.g. “New: ·It joins ·It at the baseline (the old pipeline broke there) — decided by qsIt.yaml policy.extend[0] (join-count rank).” When that line is not enough, the Why? button (or `x`) opens the full panel:

- **Explain** is the candidate table the settlement function considered at each divergent position: every surviving candidate with its entry/seam and join-count, `->` marking the winner, the eliminated candidates each attributed to the YAML record that eliminated them, and a `decided by:` line naming the stage that separated the winner from the runner-up (join-count rank, declaration order, a prefer, or the structural floor).
- **Provenance** lists the `glyph_data/runes/*.yaml` records the outcome was attributed to — these are the levers a reject’s policy draft will name.
- **Drafts** previews the three precomputed verdict artifacts (pin / policy edit / any-of) this unit would export.

### Triage flow

- The sidebar lists the ledger classes with the ledger’s verbatim `why`/`status` text; clicking one filters to that class. Two accent outlines keep it oriented, echoing the main pane’s cursor ring: every class with units in the current batch (or worklist) carries a lighter-shade one, and the class holding the cursor unit the full cursor blue — independent of the border marking a clicked class filter. Units within a class are grouped by family pair (fold/unfold per group), ordered by codepoints, and paged in batches of 300 (`[` / `]`). The progress counters run big-chunk → little-chunk left to right: the header strip leads with the whole-surface unit total, then the machine-approved count split by channel (ink-identical + junior-equivalent), then the no-verdict-class exemptions, then the human-workload Overall progress with its per-verdict tallies, then the selected class’s progress (when a class filter is active), then the current batch — classes and batches cross-cut (a batch can span classes and a class can span batches), but selecting a class narrows the batch list, so class-then-batch is the drill-down as experienced. Each sidebar entry’s counts line follows the same order: class units, machine-approved, verdicted/human, rows. Verdict-progress numerators and denominators count the human workload only.
- Machine-approved units (ink-identical or junior-equivalent) are hidden by default. The toolbar’s “Show machine-approved” toggle (state in the URL hash as `machine=1`) reveals them, grouped by class in collapsed folds below the human units, each row carrying its channel’s inert “… — machine approved” badge and disabled verdict buttons whose title explains why no human input is meaningful (ink-identical: both fonts render the unit identically; junior-equivalent: the rebuild’s ss10 rendering is the Junior font’s isolated rendering, which is the ratified ss10 spec). Deep-linking to `#unit=<an ink-identical id>` with the toggle off reveals just that one unit transiently — the persistent toggle stays off, the URL stays clean, and navigating away hides it again. Keyboard flow, auto-advance, and `g` group-approve operate over the human workload only.
- All view state (class, batch, cursor unit) lives in the URL hash — reloading or sharing the URL restores exactly where you were. Verdicts autosave: every store mutation debounce-POSTs the full export payload to the serve script’s `/autosave` endpoint, which atomically writes `verdicts-autosave.json` at the repo root (outside `rebuild/out/review/`, so livereload’s JSON watch doesn’t turn each save into a page reload), and the app restores a stamp-matching autosave on load — a reload, crash, or closed tab no longer loses in-progress work. An autosave whose `manifest_generated_at` differs from the live manifest is never joined by unit id: the app leaves it unrestored (a toast says so), and the server stashes it aside as `verdicts-autosave-<stamp>.json` before the first overwrite, preserving pre-rebuild work for a content-keyed carry. The unexported nag (the beforeunload warning and the every-50 toast) only fires when autosave isn’t working — e.g. when the surface is served by something without the endpoint — and the status bar’s unexported count gains an “(autosaved)” suffix when it is.
- **The docket view** (the header’s Docket button, `#view=docket`) is the live work order over the blank units: it groups every blank (unverdicted or skipped) human unit by the build-emitted `cluster` signature — the echo key minus the judged pair, so one visual question recurring across letter contexts is one card — and lays out the queue largest-decision-first: a top-25 tranche of cluster cards with exemplar render pairs and evidence from judged lookalikes, later clusters as a compact table, singletons in 40-unit chunks, and every echo group whose recorded verdicts disagree. Every button is a `#units=` worklist link, so judging happens in the normal keyboard flow with echo-fill, and the view recomputes from the in-memory store as verdicts land — clusters shrink, empty, and promote with no re-bake. Blank units in ledger-ruled classes are excluded behind a one-line count (a class-level decision or bulk-proposal import covers those), and any filter or class click exits the view. `rebuild/tools/review_docket.py <verdicts-file>` writes the same clustering frozen against a verdicts file to `tmp/docket-data.json`, the data feed for bulk-proposal authoring. Its complaint-side sibling, `rebuild/tools/complaint_docket.py` (`make complaint-docket`), clusters the open reject/neither verdicts by the rune records that decided them and writes `tmp/complaints-data.json`, with per-group park candidates (still-blank units the same records decide) and `--park` emission of `verdicts-park-*.json` skip files for the Import dialog; `parked` joins `carried`/`echo-fill`/`echo-harmonize`/`bulk` as a recognized note-marker kind.
- Each row shows the codepoint string in both fonts on a checkered background, the divergent pair highlighted with an amber band (precomputed x-ranges in font units; the frontend never measures text), a visible tick on ZWNJ boundaries, a per-unit Copy button that yields a prompt preamble, the one-line summary sentence, and a Why? button (or `x`) opening the explain panel with the §6.3 attribution. Unmarked regions can be trusted: they render identically in both fonts, or differ only invisibly at name grain.
- The same amber marks the pair on the text lines: the notation line (·May·Oy·It·May) and the codepoints line (E665:E679:E670:E665) underline the divergent pair’s tokens in the band’s `--hot` color. The mapping is computed at build time (`pair_codepoints`, a codepoint-position span, plus `notation_tokens` aligned one-to-one with codepoint positions) because ligatures make glyph positions diverge from codepoint positions; units with no primary pair render both lines unmarked, and only the primary pair is marked — secondary seams stay out of the text lines.
- Longer units can contain divergent seams beyond the amber-banded primary pair. Each such secondary seam renders in both columns as a dimmer dashed band with a small chip naming its home unit — the shortest unit where the same before/after behavior is the primary judgment — and clicking the chip deep-links there (cross-batch, cross-class, machine-approved homes included). A chip reading “only here” means no shorter home exists (a genuinely context-dependent divergence at the depth-2 horizon): judge that seam in this unit. Seams whose home is ink-identical get no marker at all — the difference is an invisible name-grain rename with nothing to see. The visible-marker census is pinned by tests and recorded in the manifest.
- There is no config-chip strip: chips carried no information for the majority of units that diverge under all non-ss10 configs, so they were removed. Instead, a unit shows one small inert badge only when its build-time `config_note` is non-null — computed generically from the unit’s config set against the manifest’s full list as “only when f is on” (the set is exactly the configs containing feature f; phrased “only under ss10” for the isolation overlay), “only when f is off” (exactly the non-ss10 configs without f), or a literal “only under: …” fallback. That surfaces exactly the judgment-relevant cases — ss03-gated, ss03-excluded, ss10-only — which also explain the row’s `font-feature-settings`; the badge’s title lists the full config set verbatim.
- Units are deduplicated by (codepoints, baseline outcome, new outcome), so every config of a unit renders identically by construction; the build groups configs by rendered-outcome identity into `render_groups` and the tests pin the invariant that every M1 unit has exactly one group. If future data ever violates it, the app renders each extra group’s before/after pair stacked below the first with its own config label and feature settings — nothing is collapsed, nothing needs clicking.
- Verdicts always cover the whole unit (all of its configs); the per-config verdict scoping that the original chips offered is gone, and `verdicts.json` records carry no `configs` field (legacy files importing one have it ignored).
- _Export_ downloads `verdicts.json` (format `ams-review-verdicts/1`, includes notes); _Import_ merges a previous export back in by unit id (with a manifest-mismatch warning and a force path), so triage can span sessions. The autosave complements but doesn’t replace exports: `verdicts-autosave.json` is a rolling safety net scoped to the live manifest, while downloaded exports remain the durable masters that feed the export CLI.

### What to do with each exported artifact

Run the export CLI on the downloaded file; the resulting `tmp/review-triage.yaml` has five sections:

- **`pins`** (from approvals): whole-word `data-expect` strings, each already validated for syntax (the repo’s real `parse_expect`) and semantics against the after-font. Copy each into its `suggested_home` (e.g. `site/the-manual.html`) as a `data-expect` / `data-expect-noncanonically` attribute — this is the opinions-become-pins loop from §10 item 5. `duplicate_of` flags pins already covered.
- **`policy_edits`** (from rejections): one-line `policy.prefer[+]` / `policy.refuse[+]` / `policy.contract[+]` records naming the target rune file, an appendable keypath, the deciding provenance records, and the decided stage. These are _drafts_, never auto-applied: review each, paste into the named `glyph_data/runes/*.yaml`, and rebuild. Units with `no_mechanical_draft` have no expressible one-line counter-lever — start from `names_provenance` and the unit’s explain panel by hand.
- **`any_of`** (from fine-either-way): candidate expect strings (after-behavior first) for the any-of channel, realized as `_assert_expect_any` until the corpus connective exists. Units whose before/after behaviors are whole-word-identical (cell-only divergences) deliberately carry a single candidate.
- **`neither`** (from neither-verdicts): both the old and the new behavior look wrong, so nothing automatic is drafted — no pin, no policy edit, no any-of. Each entry carries the unit id, codepoints, notation, the reviewer’s note, and `names_provenance` (the `glyph_data/runes/*.yaml` records the outcome was attributed to — the follow-up author’s levers). These units need fresh authoring work, starting from the provenance and the unit’s explain panel.
- **`identical`** (from identical-verdicts): the reviewer cannot see the flagged difference — the highlighted portion looks identical. Nothing is drafted; each entry carries only the unit id, codepoints, notation, and the reviewer’s note. These are claims that the flagged divergence is invisible, which is signal for the ink-comparator and highlight tooling (a difference it flagged that no human can spot), not a judgment on either font’s behavior.

## Mechanisms

### Machine-approved units

`rebuild/review/ink.py` is the build-time gate: a unit is `ink_identical` iff shaping its text in both shipped fonts under **every** config in its set produces identical placed outlines (uharfbuzz shaping via the validation `Shaper`, always with `kern: False` merged into the config’s features via `ink.kern_neutral`; per-glyph `DecomposingRecordingPen` outlines translated by the cumulative `x_advance` plus `x_offset`/`y_offset`; pieces sorted and compared). These units get `batch: null`, are excluded from batches, sidebar counts, and progress denominators, and surface in the app only behind the “Show no-verdict units” toggle, badged and verdict-disabled. The manifest records the totals (the `machine_approved` totals, the method one-liner, per-class counts), the export CLI reports them as a `machine_approved` section in the triage YAML (counts, per-class counts, method, compact unit-id ranges — never as drafted pins), and `verdicts.json` stays human-verdicts-only. The numbers are pinned by tests and must reproduce exactly on a rebuild; in table-diff mode the same comparison runs over each entry’s witness string, and a witnessless entry stays in the human workload because there is nothing to shape.

### Junior-equivalent units

The second machine-approval channel, added when the blessing pass hand-approved ~2,400 ss10-only units whose correctness never needed a human: `ink.JuniorOracle` approves a unit divergent **only** under ss10 (the suppress-all-joins set, whose ratified meaning is fully isolated letters — see the ss10 entries in `rebuild/m1-divergences.yaml`) when the rebuild's ss10 rendering places exactly the ink the shipped Junior font places for the same string, once Junior's letter tracking is removed. The premise — Junior carries the same isolated letterforms as Senior plus exactly one pixel of extra advance on every Quikscript glyph, and no advance difference anywhere else — is verified against the shipped Senior at oracle construction, which refuses to run if the fonts drift from it. A pass means the rebuild draws the canonical isolated rendering, so the new behavior is the spec by construction regardless of what the old font did (the old font variously kept joins, formed ligatures, or kerned under ss10). Units also divergent under any other config are never oracle-approved: the other legs still need judgment, and verdicts are per unit. `junior_equivalent` flows through `audit.Unit` and the unit JSON exactly like `ink_identical` (`batch: null`, its own `junior-equivalent — machine approved` badge, counted in `machine_approved_count` and the manifest's `machine_approved.channels.junior_equivalent` record alongside the Junior font's source and sha256). On the 2026-07-04 surface the oracle agreed with all 2,391 recorded human approvals of ss10-only units and refused exactly the five `· ◊ZWNJ` boundary windows where Junior lowers the namer dot and the rebuild's ss10 run draws the plain one.

### No-verdict ledger classes

The class-blanket exemption channel, added when the ratified boundary-equals-word-boundary rule left non-ink-identical `boundary-echo` units sitting in the verdict queue with live buttons: a ledger entry may carry `no_verdict: true`, meaning a ratified rule adjudicates the whole class and no unit in it ever needs an individual verdict. The flag flows from `rebuild/m1-divergences.yaml` through `audit.LedgerClass`/`Unit` into every unit and class record; `assign_batches` gives exempt units `batch: null` exactly like ink-identical ones (the §7 checker enforces both), the app treats `ink_identical || no_verdict` as “needs no verdict” (out of the cursor queue, controls disabled, hidden behind the toggle, `no verdict needed` badge, sidebar progress replaced with the badge), and `export.py` skips any verdict record landing on an exempt unit (`skipped_no_verdict` in the triage counts) so stale masters can’t smuggle boundary pins into a triage YAML. Only `boundary-echo` carries the flag today; extending it to another class is a one-line ledger edit.

### Ink-duplicate unit merging

The third dedup channel, added when the blessing pass caught the surface asking the same visual question twice: a unit (e.g. ·It·Day·Tea·No, “only when ss04 is on”) can be pixel-identical to its default-config sibling in both fonts — the old font’s ss04 lookups rename word-initial ·It (`qsIt.en-y0.ex-y0.before-day` vs `qsIt.ex-y0.before-day.after-no-baseline-join`) without moving any ink, and the name-grain `(codepoints, baseline, new)` dedupe key splits the window on the label alone. `audit.merge_ink_duplicate_units` (called by `build_m1` between `load_workload` and the ink-identical census) folds sibling units of the same window whose `InkComparator.signature` — the placed-outline pair for (before, after) — is identical across every config on both sides. The earliest-config sibling survives with the union of configs, rows, kinds, and `config_classes`, keeps its own name tuples for display, re-resolves its class UNMATCHED-wins, and collapses to a single render group (ink identity is exactly render-group identity, so the one-render-group invariant survives). A fold that would put two distinct matched ledger classes on one unit is skipped — different names legitimately hit different ledger predicates. Because `assign_family` runs post-merge, the merge also pulls relabeled-only ss04 halves out of `deferred-ss04` into the default families where their real adjudication lives, so they are never re-asked in the deferred pass.

Two safety consequences. First, the manifest stamp: `_generated_at` derives from input mtimes, and this was a code-only change, so without intervention the renumbered surface would have kept the old stamp and let a stale autosave or export attach by id to the wrong units. `build.UNIT_ASSEMBLY_EPOCH` now floors the stamp; bump it whenever a build-code change re-keys or renumbers units with no input change. Second, verdict carry: `rebuild/tools/carry_verdicts.py` grew `--source SURFACE_DIR VERDICTS_JSON` (repeatable) and `--out`, excludes `secondary_seams` from the content key (its `home` field embeds another unit’s id, which churns on every renumbering), and falls back to matching by `(codepoints, ink signature)` — the merge criterion itself — for verdicts whose pre-merge content keys no longer exist. When a merged unit’s halves carry conflicting prior verdicts (the approve-vs-neither contradictions on ink-identical siblings are the double-ask pathology this whole mechanism retires), neither is carried and the unit is reported for one fresh look.

### Echo groups — one verdict per rendered change

The dedup channel one grain above the ink-duplicate merge: different windows whose before→after change is the exact same picture. The merge handles identical whole windows; echo groups handle the far larger population where the _change_ is pixel-identical — the same pieces of ink appearing and disappearing, the same judged letter pair, the same class and config set — but the surrounding letters differ (·Pea·Day·Tea·No, ·May·Day·Tea·No, ·It·Day·Tea·No, … all showing the identical ·Day·Tea regrouping). The grain is (configs, judged pair, class, normalized ink delta).

Build side: `InkComparator.config_diff` returns the per-config before→after ink delta (the multiset difference of placed pieces, jointly normalized so the delta’s leftmost point sits at x=0); the same pass that computes `ink_identical` stashes it, and after family promotion every human unit gets an `echo: e-NNNN` group id (exempt units carry null; the §7 checker enforces both, and `totals.echo_groups` pins the count). Echo ids are order-derived and excluded from the carry content key, exactly like `secondary_seams`.

App side: a unit in a multi-member group shows an “echo ×N” chip (click = the group as a `#units=` worklist, stacked), and typing a group id (e.g. `e-0061`) into the cross-class search offers the whole group stacked the same way. A verdict on any member — button, key, canned reject/neither note, or `g` group-approve — fills the _unverdicted_ rest of the group with the same verdict and note as one undo step, with a toast reporting the fill; `skip` never echoes, already-verdicted members are never overwritten, and any member can still be individually overridden or cleared afterward (the store records stay per-unit, so export, import, autosave, and carry are all unchanged).

Seeding and auditing: `rebuild/tools/echo_verdicts.py <verdicts-file>` emits `verdicts-echo-fill.json` (fill records for blanks in unanimously-verdicted groups) and prints every group whose recorded verdicts disagree, each with a ready `#units=` deep-link.

### The artifact cycle mechanized

The commit-time artifact cycle — snapshot the served surface, `run_m1`, rebuild the surface in place, carry the verdicts forward, re-baseline the census pins, run the three gates — used to be re-derived from prose and orchestrated by hand each time. It is now one command: `make artifact-cycle` (the driver is `rebuild/tools/artifact_cycle.py`). The driver refuses to start while a review server is listening on port 7294 (an in-place rebuild would strand the live session’s in-tab store and orphan its autosave across the manifest restamp; `--yes` overrides with a warning), and snapshots `rebuild/out/review` to `tmp/review-pre-<short hash>` before any mutation (the carry input and the only recovery copy). When `--verdicts` is omitted it auto-resolves which master to carry — the stamp-aligned candidate with the most effective verdicts among the autosave and the `verdicts-*.json` files at the repo root and under `rebuild/evidence`, falling back to the newest-stamped candidate when nothing aligns (carry re-resolves by content and ink keys, so a stale stamp is survivable), and proceeding without carry when no candidate holds a single effective verdict — always printing what it picked; `--verdicts` names a master explicitly and `--no-carry` skips the step. Because `run_m1` exits nonzero whenever any oracle rows are UNMATCHED — always true mid-migration — the driver gates on the four summary JSONs instead of the exit code: it aborts only on defect errors, a boundary-gate failure, a Manual-pin failure, or `multi_matched > 0`. The gates run sequentially (JS suite via the `node --test rebuild/review/jstests/*.test.js` glob form, the rebuild suite with the four documented batch-1 spec pins subtracted as the expected baseline, then `make test`), a summary table always prints even on failure, and `--dry-run` shows the fully resolved plan without executing anything.

### Census pins as one reviewable file

The census counts the review tests pin — previously hardcoded literals scattered across `rebuild/test_review_{audit,build,ink,families}.py` and re-baselined by hand-editing four files after every rebuild — now live in the checked-in `rebuild/review-census-pins.json`, loaded by the tests through `rebuild/review/census.py`. The same module is the regenerator (`uv run python -m rebuild.review.census`): `--check` (the default) recomputes everything and exits nonzero with a per-key report when the pins are stale; `--update` rewrites the file; `--surface DIR` reuses an already-built surface for the manifest-derived groups (the driver passes `--surface rebuild/out/review` right after rebuilding it), and without it the tool builds a fresh temporary surface so it can never describe a stale one. Because tests and regenerator call the same computation functions, they cannot drift. The file is grouped by source and grain — `manifest` and `built` are post-merge (the shipped surface), `audit` and `ink` are the pre-merge name grain, `families` is the triple grain (a class count there can differ from the post-merge manifest, which is the proof the manifest cannot source it). The pins remain a tripwire, not a self-blessing snapshot: regeneration is explicit (`--update-pins` on the driver, which prints the resulting `git diff` in full), and re-baselining is now “review one diff”, with the judgment — does every moved number match the commit’s expected scope? — intact and in one place. Semantic pins deliberately stay hardcoded in their tests: the drafts worked-example window, the enrich `diff_positions` exemplar, `FAMILY_ORDER`, and the `test_surface.py` unlock lists. `build_m1` and `build_table_diff` also now prune orphaned `units/*.json` shards after writing the manifest, retiring the clear-the-units-dir-by-hand wart.

- **Source layout**: `rebuild/review/` is a normal package — `audit.py` (TSV + ledger ingestion), `tablediff.py`, `enrich.py` (seams, divergent positions, highlight ranges, explain attribution, all precomputed at build time), `drafts.py` (the three verdict drafters), `build.py`, `export.py`, `serve.py`, `static/` (the five ES modules + CSS + HTML shell), `jstests/`. Tests live flat as `rebuild/test_review_*.py` plus `node --test` over `jstests/`.
- **Output contract**: `rebuild/out/review/` is fully self-contained — `index.html`, app JS/CSS, `manifest.json` (format `ams-review-manifest/1`, per-class counts and sha256-recorded font copies), `units/<class>.json` shards lazily fetched per class, `fonts/before.otf` + `fonts/after.otf`. The manifest plus shard schema (REVIEW-PLAN.md §7) is the only frontend↔engine interface, and the build runs its own contract self-check. Builds are byte-identical run to run (modulo `generated_at`).
- **Future migrations**: `build.py --mode table-diff --baseline <dir> --new <dir>` plus the `snapshot` subcommand generalize beyond the M1 audit TSV, so M2+ reviews reuse the same app unchanged.
- **At cutover**: the app moves out of `rebuild/out/` and (probably) the serve script merges into `tools/serve.py`’s livereload. The user-sanctioned `review-build` / `review-serve` Makefile targets already exist (plain `review` was taken); no `site/` changes, no existing pipeline files touched.
- **Known leftovers, not blocking**: a `{"type":"module"}` `package.json` is copied verbatim from `static/` into the output (harmless); `rebuild/review/fixtures/` and `rebuild/review/jstests/fixtures/` are divergent copies that should converge on one source of truth before cutover.

### Policy drafters

`drafts.py` picks one of four counter-levers per rejected unit:

- **`policy.contract[+]`** — a `contract by: N` lever, drafted only when the gained extension rides a join both fonts share, with the `when` window following the extension’s side.
- **`policy.refuse[+]`** — when the divergence includes a join the baseline didn’t have, a `contract` cannot restore the old break, so the drafter detects the new-join side against the baseline seam and refuses the anchor reaching across that gap, scoped to that neighbor (e.g. `{exit: x-height, when: {right: {family: [qsIt]}}, …}`).
- **`policy.prefer[+]` with `mode: absolute`** — on seam-identical units, where a refuse would break joins both fonts share; it pins the baseline cell (or stance) over the new anchors.
- **`no_mechanical_draft`** — adjustment-grain units (post-ZWNJ locked twins, bind pullbacks, non-summing-suppressed extensions) have no expressible one-line counter-lever and carry no policy draft; `export.py` still emits them in `policy_edits` with null fields, the unit’s provenance, and a `no_mechanical_draft` explanation, so rejected units never vanish from the triage YAML.

### Verdict families (UNMATCHED presentation)

The round-3 surface presents the **UNMATCHED** verdict windows the old surface dropped — a presentation-only grouping that leaves the oracle untouched until the families are adjudicated.

- **Per-config class.** A render triple `(codepoints, baseline, new)` no longer maps to one ledger class: a window can be blessed under one stylistic set yet novel under the default config. `audit.py` keeps the window as one unit, records every config’s class in `config_classes`, and adjudicates the novel (default) behavior (UNMATCHED-wins); the shard carries a `config_class_note` blessed/novel strip the app renders as a badge. A two-_matched_-class triple still raises as a real bug.
- **Verdict families.** `families.py` groups the UNMATCHED triples into presentation-only synthetic classes (`status: unmatched`), appended after the ledger classes so they sort last and clean-unit ids stay stable. The grouper reads the before/after settled seams from `enrich` — a gain (`break → yN` or a raised seam) names a family by the joining pair (the seam is the left cell’s exit, i.e. a ligature’s trailing component, joining the right cell’s entry); every loss pools into `seam-loss-withdrawal`; an unchanged-seam/changed-cell window is `extension-non-summing`; anything else is the `unmatched-misc` catch-all (the grouper is total). Stylistic-set-only windows are parked in `deferred-ss04` / `deferred-ss10` / `deferred-ss03`, sorted last.
- **Enrich fix.** `after_cells` is the re-derived settled cells, not the audit `new` column — the `kern-channel-out-of-scope` position-only rows carry per-slot diagnostics there, not cell tokens, so the old reading broke the `after_seams`-parallel-`after_cells` contract once the build completed over the full audit.

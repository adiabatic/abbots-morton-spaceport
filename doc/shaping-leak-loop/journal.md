## General lessons (read before picking a target)

- **A family-level `not_before: {family: qsY}` on a left glyph's default baseline-exit form breaks the real qsX→qsY baseline join.** Many backlog "single exit edge before bare follower" entries (e.g. `She No Gay Out`, the `qsGay.ex-y0`, `qsMay.ex-y0`, `qsJai.ex-y0` lines) are _default-exit dangles_: the left glyph carries its normal baseline exit because something follows in the run, but across the break the follower is shaped bare (entry y5) so the exit dangles. The catch — the same family also has a real baseline-entry variant (`qsNo.alt`, `qsZoo.half`) that the left glyph genuinely joins to in connected text, and a blanket `not_before: {family}` suppresses the exit for _both_ the break and the real join, so `make test` breaks `qsX→qsY.<baseline-entry>` joins. The leak gate (`make test-leaks`) will still pass — it doesn't see real joins — so **you must run `make test` for any exit-suppression lever before trusting it.** A correct fix needs to scope the suppression to the break/non-join context only (the bare follower form), not the whole family; a coarse family `not_before` is the wrong lever for these.

- **Productive pattern (criterion A — non-joining neighbor):** the clean wins are exit/entry dangles toward a neighbor the changed glyph _never_ joins in any context. Prove it from the design itself: check the neighbor's entry/exit form `not_after`/`not_before` already excludes the changed family, and/or grep the generated FEA (`test/AbbotsMortonSpaceportSansSenior-Regular.fea`) to confirm no real-join rule produces the receiving form after that predecessor. Often a _sibling_ form of the changed glyph already carries the exclusion (e.g. May's `entry_xheight_exit_baseline.not_before` listed qsHe but the bare `exit_baseline` didn't) — adding the missing family to the existing `not_before`/`not_after` just closes an inconsistency and is low-risk. This is the opposite of the lesson-1 trap (where the neighbor _does_ have a matching-Y join variant). Still run `make test` to be sure.

- **Refinement to criterion A (entry must not be load-bearing):** before adding a plain `not_before: {family: qsNeighbor}` to suppress an exit dangle, confirm the form you are suppressing does NOT also carry an ENTRY that the changed glyph's own _predecessors_ join into. The May→He win worked because `qsMay.exit_baseline`'s entry side wasn't load-bearing in that context. The No→He attempt FAILED (12 new bad leaks) because the suppressed form `qsNo.alt` has a baseline `entry: [0,0]` that ·No's predecessors (·It/·Gay/·Roe/·They/·Vie/·Ye/·Utter/·Jai/·Excite) genuinely join into — suppressing `alt` before ·He removed that entry in the "…No He" context, collapsing every predecessor's real join into a fresh dangle. So the test is: does the promoted form gain BOTH an entry and an exit (e.g. `qsNo.alt.en-y0.ex-y0`, `qsDay_qsUtter.half.en-y0.ex-y5`)? If yes, a plain `not_before`/`not_after` is the wrong lever — it reroutes the backward-upgrade chain (lesson 2). Such a target needs an entry-PRESERVING `ex-noentry` form (`inherits` the promoted form, `anchors.exit: null`, `before:`/context-scoped to the non-joining follower) so predecessors keep their entry while the dangling exit is dropped — that is second-order authoring, not a one-liner. Prefer targets where ONLY the break-facing edge changed (`qsMay.noentry->qsMay.ex-y0` gained only an exit; that was clean).

- **Inflection point (one-liner seam ~exhausted after fix [1]):** an exhaustive criterion-A sweep (iteration 4) found NO remaining clean one-line `not_before`/`not_after` fix of the May→He shape. ·He was structurally unique — the only universal non-receiver (no x-height entry; `qsHe.entry_baseline.not_after` excludes ~everyone). Of the remaining ~194: ~30 changed forms gain BOTH an entry and an exit (`qsGay.en-y0.ex-y5` ×12, `qsUtter.alt.en-y5.ex-y0.reaches-way-back` ×6, `qsNo.alt.en-y0.ex-y0`, `qsRoe.en-y0.ex-y5`, `qsOut.en-y0.ex-y5` …) → load-bearing-entry traps (lesson 3); the exit-only/entry-only promotions (`qsMay.ex-y0`→Gay, `qsJai.ex-y0`→It, `qsGay.ex-y0`, `qsShe.ex-y0`→No, `qsHe.half.ex-y5`→It/Jai, `qsWay/qsWhy.half.ex-y0`→Gay, `qsOwe.ex-y5`, `qsTea.half.ex-y5`) all dangle toward a neighbor that DOES receive the reach in some real-join context → lesson-1 traps needing break-context-scoped suppression, not a family selector. Net: the remaining backlog needs SECOND-ORDER machinery (entry-preserving `*.ex-noentry` forms, break-scoped `before:`/`after:`, demote-overrides) — the same class the runbook defers to the hard track. Proof basis: see iteration-4 sub-agent file:line citations (May→Gay FEA :3332; Roe→No explicit `before:{family:qsNo,entry_y:0}`; It/Jai x-height entries :2584/:1411). **Confirming deep sweep (also iteration 4):** the last uncleared exit-only families — qsOwe.ex-y5, qsTea.half.ex-y5, qsPea.half.ex-y5.ex-dips, qsWhy.ex-y5, all reaching x-height toward qsIt/qsJai/qsYe (backlog L115-119, L132-133, L192-193) — are ALSO real x-height joins (rendered with `tools/inspect_join.py`: every pair completes at y=5; qsIt `entry_xheight.not_after` accepts them, qsPea even courts qsIt/qsJai via `before:` at :334-335, qsJai `contract_entry_after` names qsTea/qsHe/qsPea at :1415).

The user has just adjudicated units in the review app and wants the fresh complaints landed as rune changes under `glyph_data/runes/`. The design decisions are theirs (recorded in verdicts and notes); your job is to get from "18 fresh complaints" to "the one or two record changes they imply" without re-deriving the repo's invariants. Batch precedent: 94 rejects → 2 rune changes; 17 rejects → 1. If you are proposing more than ~2 changes per flaw cluster, you have probably mistaken fix sites for fixes.

## Hard rules

- Fresh complaints are the work; **standing complaints stay standing unless the user asks** — they are usually tied to an open fork recorded in WHATNEXT.md. Check there before touching one.
- Never write or edit the verdict store or any `verdicts-*.json`, never run `merge_verdicts` while the review server is up, and never rebuild the review surface (`review.build`) — surface rebuild, carry, and merge are `make review-cycle`'s job, which the user runs after the fix commits.
- Never author `why:` text. Lift the user's verdict note verbatim (the `note`/`gist` on the reject unit) onto the one record it actually describes — not onto sibling guard records it doesn't. Where the schema has no `why:` slot (unlocks), the verbatim note goes in a `#` comment on the record instead.
- Never commit without explicit approval; at the natural commit point, spawn a fresh sub-agent for commit-message suggestions (reader-experience phrasing, not mechanism).

## Steps

1. **Bake the docket.** `uv run python rebuild/tools/complaint_docket.py` (or `make complaint-docket`; defaults to the live autosave) → `tmp/complaints-data.json`. Groups sharing pointer bases (`shares_pointers_with`) are **one flaw seen from multiple mechanical fix sites** — the `suggested_records` drafts are grouping keys, not the fix, and a refuse-shaped draft usually just moves which seam is sacrificed. Collapse to flaws before designing anything.
2. **Read the ground truth.** The definitive join map is `rebuild/out/m1/baseline-<config>.subset.tsv.gz` — grep the flaw's codepoint windows across configs (never scope from FEA reconnaissance; it misses bare-carrier joins). The unit JSONs under `rebuild/out/review/units/*.json` carry each window's `before`/`after` renders and an `explain` trace naming the deciding record and every eliminated candidate — read the explain before reading rune files.
3. **Map every config arm.** The same codepoint window is several units (the ss03-off arm, the ss03 arm, sometimes ss10 rides a ledger-ruled class instead). Pull all sibling units and their verdicts from the autosave before designing: **an approved arm must not move.** Where a fix would spill into an approved arm, guard it — refusals and unlocks are pure unions, so a carve-out is expressed by restructuring the refusing record (split the family out of the broad list; add an `except:`'d record plus `feature:`-scoped records restoring the old behavior under the sets that must not change).
4. **Design the minimal record change.** The closed vocabulary is design §3.3/3.4; the gotchas that cost time: `except:` entries are static atoms (family/class/stance/trait — **no `joined_at`**, so "joined at height H" carve-outs must be argued extensionally); a `left:` of only `except:` matches boundary lefts too (a run-splitting token makes the left a boundary, not the letter behind it); refuse/require may not use `right.then`; unlock records take no `why:`; stance `order:` breaks join-count ties, which is often what keeps the fourth-letter windows (·Tea·Day tight bond and friends) stable for free.
5. **A/B with probe.** `PYTHONPATH=. uv run python rebuild/tools/probe.py E6XX:E6XX:…` renders old-font vs live-runes settlement across all 8 configs. Before editing, probe the flaw windows; after editing, run a battery over every rejected window plus the must-not-change neighbors (the bare pair, other left/right families, boundary-token flanks), printing only old-vs-new divergences — every ss03-off row should go silent, and every surviving divergence should be a pre-existing adjudicated one (confirm by matching the stored unit's `after` render byte-for-byte).
6. **Gates.** `uv run python -m rebuild.pipeline.run_m1 --jobs 8` (defects 0/0, budget yellow-not-failed, boundary and Manual pins pass; the oracle-unmatched delta is the score — fixed windows leave the pile, deliberate new divergences join it under existing classes). Then `--conform-only --jobs 8` (must stay exact) and `uv run pytest rebuild/ -n auto --dist worksteal` — never single-threaded. Diff the failure set against the expected buckets in the current batch progress file: baseline spec-pins plus census-pinned tests (stale until the next cycle) are expected; a genuinely new failure is usually a **content pin** your change legitimately moved — update it by hand. `make test` self-skips; `make prettier` after any Python change.
7. **Land.** `uv run python tools/reflow_yaml.py` on the touched runes (expect a no-op), confirm the spec loads, show the diff, present commit-message candidates, wait for the go-ahead. After committing, add the one-line-per-commit entries to the batch progress file and remind the user the next step is `make review-cycle` — the churn forecast's stranded approvals re-queueing is expected, not a defect.
